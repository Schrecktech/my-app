# Internationalization (i18n) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extract all hardcoded user-facing strings into a centralized i18n system using `expo-localization` + `i18n-js`, shipping English only.

**Architecture:** Device locale is detected via `expo-localization` at startup. Strings are stored in TypeScript locale files under `constants/locales/`. Components call `i18n.t('namespace.key')` instead of hardcoding strings. Fallback to English for any missing translation.

**Tech Stack:** `expo-localization`, `i18n-js`

**Spec:** `docs/adr/0009-internationalization.md`

```ts
// constants/i18n.ts
import { getLocales } from 'expo-localization';
import { I18n } from 'i18n-js';
import en from './locales/en';

const i18n = new I18n({ en });
i18n.locale = getLocales()[0]?.languageCode ?? 'en';
i18n.enableFallback = true;

export default i18n;
```

```markdown
## Adding a New Language

1. Create `constants/locales/<lang>.ts` — copy `en.ts` and translate the values
2. Import and register in `constants/i18n.ts`: `const i18n = new I18n({ en, es })`
3. Done — devices set to that language will use it automatically
4. Missing keys fall back to English — partial translations are safe to ship

## Adding a New Translatable String

1. Add the key and English value to `constants/locales/en.ts` in the appropriate namespace
2. Use `i18n.t('namespace.key')` in your component
3. Add translations to other locale files (optional — English fallback is automatic)
```
